## Структура документа (главы)

Парсер ожидает нумерованные главы (`## 01. …`, `## 14. …` и т.д.).

| Главы | Роль | Индекс FAISS | Runtime |
|-------|------|--------------|---------|
| **00** | Мета-политика, дисклеймер | Нет | В RAG system prompt |
| **01–12** | Операционные SOP | Да (`sop`) | Векторный поиск |
| **13** | Out-of-scope | Нет | В RAG system prompt |
| **14** | Центральный FAQ | Да (`faq`) | Векторный поиск |
| **15** | Глоссарий | Нет (отключено в MVP) | — |
| **16** | Деревья решений | Да (`decision_tree`) | Поиск + walkthrough |
| **17** | Сценарии | Да (`scenario`) | Векторный поиск |

### Извлечение FAQ

Чанки FAQ из:

1. **Главы 14** — центральный FAQ.
2. **Блоков FAQ** в конце SOP-глав (01–12).

В каждом FAQ-чанке — метаданные `[Источник: <глава>]` для trace.

---

## Рекомендации по типам контента

### SOP (главы 01–12)

- Одна процедура на логический раздел.
- Заголовки `###` для шагов.
- Действия в повелительном наклонении.
- FAQ в конце главы для типовых вопросов.

### FAQ

- Пары вопрос–ответ.
- Кратко, со ссылкой на SOP по смыслу.
- Дубли между главами — лучше в гл. 14.

### Decision trees (глава 16)

- Явные условия ветвления.
- Критические ветки и эскалация.
- При similarity ≥ 0.30 — **отдельный операционный walkthrough**; пишите как пошаговый алгоритм.

### Сценарии (глава 17)

- «Что произошло → что делать».
- Та же терминология, что в SOP.

## Стиль

| Делать | Избегать |
|--------|----------|
| Единая официальная терминология | Противоречия между главами |
| Явная эскалация («вызвать руководителя») | Размытых «действуйте по ситуации» |
| Даты версий в заголовках | Устаревших кодов без контекста |
| Русский (язык демо KB) | Смешения языков в одном чанке |

Язык UI (RU/EN) независим от языка KB.

## Главы 00 и 13 (статический контекст)

**Не индексируются.** Загружаются в runtime в RAG system prompt:

- **00** — принципы продукта, консультативный характер.
- **13** — темы отказа или перенаправления (HR, юристы и т.д.).

Полный текст на каждый RAG-запрос (без суммаризации в MVP).

## Процесс обновления

```mermaid
flowchart LR
    A["Правка rag-document.md"] --> B["Review в git PR"]
    B --> C["make etl-ingest"]
    C --> D["make etl-stats"]
    D --> E["Smoke-test в RAG"]
    E --> F["Обновить golden set"]
```

| Шаг | Ответственный | Действие |
|-----|---------------|----------|
| 1 | Владелец KB | Правка markdown; compliance для реальной KB |
| 2 | Dev / CI | Ingest (инкрементально) |
| 3 | QA | [rag_evaluation_ru.md](rag_evaluation_ru.md) |
| 4 | Ops | Бэкап `backend/data/` перед крупными правками |

### Инкремент vs полный rebuild

| Изменение | Команда |
|-----------|---------|
| Текст, новые разделы | `make etl-ingest` |
| Смена embedding model | `rebuild=true` |
| Смена логики chunker (деплой кода) | Полный rebuild после деплоя |
